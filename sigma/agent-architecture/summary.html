已掌握概念

① 纵深防御
90%
5层防御：安全（Prompt约束、输入清洗、SQL验证）vs 鲁棒（表名修正、异常隔离）。Agent模式下安全与鲁棒的边界是动态的。
② result_store 模式
85%
闭包注入字典作为侧通道，解决Agent黑盒中捞取中间产物的问题。LLM是过度配合的执行者。
③ ReAct 循环
85%
Thought→Action→Observation自主循环，无tool_calls时自然终止。调用者视角看不见LLM内部推理。
④ Tool Schema 设计
90%
@tool把Python签名翻译为JSON Schema。参数名/类型/docstring三层沟通渠道。trade-off：少参数vs多参数。
⑤ Agent 终止条件
85%
自然终止、max_iterations、max_execution_time三层兜底。终止后应返回自然语言，错误留日志。
⑥ Chain vs Agent 决策
85%
固定路径用Chain（便宜、可预测），不确定路径用Agent（鲁棒、灵活）。工具粗粒度减少LLM调用。

学员关键洞察

Agent 本质 = 面向对象
"手写路径是线性的⋯ AI Agent 就是面向对象的"——Agent 给 LLM 一组方法（Tool），让它在 ReAct 循环里自主编排调用路径。
安全 vs 鲁棒的重分类
把纵深防护五层重新分为"防攻击"（安全）和"防幻觉"（鲁棒），指出 Agent 模式下二者边界是动态的。
Tool 设计的黄金法则
没有合理默认值的参数必须 required；参数名是沟通渠道；能不出现在 Schema 里的复杂性就不暴露给 LLM。
代码即证据
能在项目代码中定位每个机制对应的文件/行号，指出当前实现缺少 max_iterations 是设计缺陷。

# Feature: RAG 知识库核心模块

## Problem Statement

现在的系统中有好几个智能问答功能, 但没有RAG, 现在我们需要搭建一个RAG来让这些智能问答功能调用, 来增强这些问答功能

## Success Metrics

- 上传一个 10KB 的 TXT 文件后，在 3 秒内返回切片预览，预览的每个切片内容与原文对应段落一致、无乱码、无遗漏
- 确认入库后，后端日志逐条输出切片进度，格式为"第X片, 共Y片 正在向量化/已存入向量库"，每秒至少输出一条
- 对一个已知库内文档内容的查询，检索结果 Top 3 中至少包含 2 个语义相关的正确片段
- 检索返回的每个片段携带唯一标识，可通过该标识定位到它在原始文档中的位置
- 新增一种向量库实现后，不修改任何现有的检索流程代码即可正常工作

## User Stories

- 作为开发者，我上传一个 TXT 文档后，可在正式入库前预览切片结果，确认文档被正确解析
- 作为开发者，我调用检索接口时，传入查询文本即可获得排好序的相关片段，无需关心底层检索策略
- 作为用户，我与智能体交流时涉及知识库中的内容，智能体能引用原文片段给出有据可查的回答

## Acceptance Criteria

- [ ] 上传 TXT 文件后可预览前 5 个切片，切片内容与原文一致、无乱码
- [ ] 确认入库后，后端日志按"第X片, 共Y片 正在向量化/已存入向量库"格式逐条输出
- [ ] 对文档中的已知内容发起检索，返回的 Top 5 结果中与其语义相关的结果排在最前
- [ ] 检索结果中每条数据携带 chunk_id 和 source 信息，可追溯到原始文件和切片位置
- [ ] 用一个新的向量库实现替换 Chroma 后，检索功能正常运行
- [ ] 未导入任何文档时发起检索，返回空结果，不报错

## Non-Goals

- 本期不解析 TXT 以外的文档格式（PDF/Word/Excel 等）
- 本期不涉及 Milvus 或其他向量库的接入（仅用 Chroma）
- 本期不做前端知识库管理页面（文件上传通过 API 验证，UI 后续作为独立模块补充）
- 本期不修改各智能体的 System Prompt 或对话逻辑（仅提供检索接口，对接由各智能体独立完成）
- 本期不实现多文档批量导入或增量更新（每次入库全量重建索引）

## Constraints

- 向量库必须通过抽象接口对接，业务代码不得直接依赖 Chroma SDK 中的类型或函数
- RAG 模块不可用时不得影响现有智能体的对话流程（检索失败应返回空，不抛异常）
- 不得引入额外的外部中间件或服务（如 Elasticsearch、Redis）来实现检索功能
- 切片过程需保持原文完整性，不允许对文本内容做改写、截断或摘要
